# PRD: Token Analytics Dashboard

## Introduction/Overview

Token Analytics Dashboard is a React application built with TypeScript to visualize token usage data produced by the `token-performance-tuning` workflow. The app will read the exported JSON artifacts generated by the nightly automation and present the most relevant usage, cost, and efficiency signals through charts and tabular drill-down views.

The main objective is to make the current token analysis output understandable at a glance, while preserving access to session-level detail for investigation. The product must support the current JSON schema, including summary data, archived monthly history, insights, and suggestions.

## Goals

- Provide a fast visual overview of token usage trends, spikes, and outliers.
- Make it easy to identify the most expensive sessions, projects, skills, and automations.
- Surface actionable optimization findings already present in the JSON exports.
- Support archived historical data without forcing the UI to load all monthly files up front.
- Handle known schema edge cases safely, including nullable daily metrics and duplicate project names across different project paths.

## User Stories

- As a developer, I want to see total token consumption and recent trends so I can understand whether usage is improving or worsening.
- As a developer, I want to identify the most expensive sessions so I can investigate prompt bloat, context spikes, and tool churn.
- As a developer, I want to compare token usage by project, skill, automation, and agent source so I can find the biggest optimization opportunities.
- As a developer, I want to review archived monthly history on demand so I can inspect older usage without slowing down the default dashboard.
- As a developer, I want to filter and inspect machine-generated insights and suggestions so I can prioritize concrete follow-up work.

## Functional Requirements

1. The application must be built with React and TypeScript.
2. The application must load and parse the current summary export from `token-dashboard-data.json`.
3. The application must support loading the archive manifest from `token-history-index.json`.
4. The application must support loading individual monthly archive files referenced by the archive manifest when a user requests archived data.
5. The application must validate incoming JSON data against a typed schema before rendering dashboard views.
6. The application must gracefully handle `daily` rows where token metric fields are missing or `null`.
7. The application must display top-level summary metrics for:
   - total tokens
   - input tokens
   - output tokens
   - reasoning output tokens
   - session count
   - archived session count
   - max single-turn input tokens
   - total function calls
8. The application must display a time-series chart for daily token usage using the `daily` dataset.
9. The application must display a time-series chart for daily context spikes using `max_single_turn_input_tokens` or the nearest equivalent field available for each day.
10. The application must display a chart comparing token usage by project.
11. The application must display a chart comparing token usage by skill.
12. The application must display a chart comparing token usage by automation.
13. The application must display a chart comparing token usage by agent source.
14. The application must display a ranked view of the highest-cost sessions.
15. The application must display a ranked view of sessions with the largest single-turn context spikes.
16. The application must allow the user to inspect a session and view its key metadata, token totals, skill mentions, insights, and suggestions.
17. The application must present the exported `insights` in a dedicated UI section that supports filtering by severity and scope.
18. The application must present the exported `suggestions` in a dedicated UI section that supports filtering by category and priority.
19. The application must present `trend_insights` in a way that links trend events to the relevant dates shown in the daily charts.
20. The application must normalize project aggregations so the UI can show both:
   - raw grouping by `project_path`
   - consolidated grouping by `project_name`
21. The application must indicate when a chart is based on partial or archived data where some metrics are unavailable.
22. The application must provide separate dashboard views for:
   - overview
   - sessions
   - dimensions (projects, skills, automations, agents)
   - archives
23. The application must support responsive layouts for desktop and mobile screens.
24. The application must render useful empty states when optional datasets are missing, empty, or partially unavailable.
25. The application must render useful error states when a JSON file cannot be loaded or fails schema validation.
26. The application must avoid loading every archive file during initial page load.
27. The application must keep the initial dashboard interactive with the current summary dataset only, then fetch archive months lazily.
28. The application must display the dataset generation timestamp so users know how fresh the dashboard data is.
29. The application must expose the current retention window and archive count from the history metadata when available.
30. The application must be structured so new JSON dimensions can be added later without rewriting core charting and data-loading code.

## Non-Goals (Out of Scope)

- Editing token-analysis data from the dashboard.
- Triggering automations from the dashboard.
- Writing back changes to skills, prompts, or automation files.
- Building authentication or multi-user collaboration in the first version.
- Supporting arbitrary JSON schemas unrelated to the `token-performance-tuning` export format.
- Providing financial cost estimation unless a future export includes explicit price data.

## Design Considerations

- The dashboard should prioritize quick scanning first and deep inspection second.
- The visual design should avoid a generic admin-dashboard look and instead feel purpose-built for token diagnostics.
- High-severity insights and high-cost anomalies should be visually emphasized.
- Archived data should be clearly marked so users understand when detail is older or partially summarized.

## Technical Considerations

- The implementation should use a client-side schema validation layer to protect the UI from malformed or partially changed exports.
- The app should include a normalization layer that converts raw export data into chart-ready models instead of binding charts directly to the source schema.
- The data-loading layer should separate:
  - current summary data
  - archive index data
  - monthly archive payloads
- The architecture should make it easy to plug in local static files first and evolve later to hosted JSON endpoints.
- The charting library must support responsive rendering, tooltips, legends, and mixed chart types suitable for time-series and ranking views.
- The UI should avoid expensive recomputation on every render for large session arrays.

## Success Metrics

- A user can identify the top three token-consuming projects within 10 seconds of opening the dashboard.
- A user can identify whether token usage increased or decreased over recent days without reading raw JSON.
- A user can open a high-cost session and review its relevant insights and suggestions in one view.
- The default dashboard loads using only the summary dataset and does not require loading archived monthly files.
- The dashboard handles the current exported dataset without runtime crashes caused by `null` or missing metric fields.

## Open Questions

- Will the app read JSON files directly from the local filesystem during development only, or should it be designed immediately for hosted static JSON endpoints?
- Should archived monthly files be loaded through user interaction only, or should the app prefetch the most recent archived month after initial render?
- Which charting library is preferred for the first implementation: Recharts, Visx, Nivo, or another existing team standard?
- Do you want a fully client-side dashboard only, or should we reserve room for a lightweight API layer later?
- Should the first version include export features such as CSV download for filtered tables?
